# 📝 ADR-006: 인증 및 유저 계정 관리 모듈 통합 및 부분 업데이트 전략

## 🗓️ 날짜
2026-03-07

## 📋 상태
**Accepted**

## 🎬 1. 배경 (Context)
* 기존에는 `UsersController`에서 회원가입을, `AuthController`에서 로그인을 담당하는 등 유저 관련 기능이 여러 컨트롤러에 분산되어 있었습니다.
* **F-04 (프로필 및 목표 설정)** 기능을 추가하면서, 사용자의 계정 생성, 인증, 정보 수정을 하나의 '계정 관리' 맥락에서 통합적으로 관리할 필요성이 제기되었습니다.
* 특히 유저가 닉네임이나 목표 칼로리 중 일부만 수정할 때, 전송되지 않은 다른 데이터가 초기화되거나 유실되지 않도록 하는 안전한 업데이트 메커니즘이 필요했습니다.

## 🏗️ 2. 기술적 결정 (Decision)
1. **모듈 및 컨트롤러 통합**: `UsersController`를 삭제하고 모든 기능을 `AuthController`로 통합하여 `/auth` 공통 경로를 통해 계정 관련 API를 제공합니다.
2. **HTTP PATCH 메서드 사용**: 리소스의 전체 교체(PUT)가 아닌 부분 수정을 지원하기 위해 `PATCH` 메서드를 채택하여 클라이언트가 변경을 원하는 필드만 전송할 수 있도록 합니다.
3. **Prisma와 Spread Operator 활용**: 서비스 로직에서 전개 연산자(`...`)를 사용하여 DTO에 포함된 필드만 동적으로 업데이트 쿼리에 반영함으로써 데이터 정합성을 유지합니다.
4. **UserResponseDto 기반 데이터 정제**: 모든 사용자 정보 반환 시 `UserResponseDto`를 거치게 하여 `goalCalories` 등 필요한 정보는 포함하되, 비밀번호와 같은 민감 정보는 철저히 제외합니다.

## ⚖️ 3. 결과 (Consequences)
* **장점**: 
    * API 엔드포인트가 `/auth`로 단일화되어 외부 인터페이스의 일관성이 향상되고 관리가 용이해졌습니다.
    * 불필요한 폴더 구조를 제거하여 프로젝트의 복잡도를 낮추고 응집도를 높였습니다.
    * 부분 업데이트 전략을 통해 사용자 실수에 의한 데이터 유실 가능성을 원천적으로 차단했습니다.
* **단점**: 
    * `AuthService`의 책임이 커질 수 있으나, 현재 MVP 단계에서는 도메인을 과도하게 분리하는 것보다 관리 효율성을 높이는 것이 더 적합하다고 판단했습니다.
