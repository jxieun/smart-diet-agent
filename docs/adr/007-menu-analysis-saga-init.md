# 📝 ADR-007: 비동기 메뉴 분석 요청 처리 및 Saga 패턴 초기화 전략

| 항목 | 내용 |
| :--- | :--- |
| **날짜** | 2026-03-08 |
| **상태** | 승인됨 (Accepted) |
| **작성자** | J |
| **영향 범위** | AnalysisModule / Infrastructure (RabbitMQ) / Database (PostgreSQL) |

---

## 📋 상태
**Accepted**

## 🎬 1. 배경 (Context)
* 사용자가 업로드한 메뉴판 이미지를 분석하는 작업은 OCR 및 AI 모델 추론이 포함되어 있어 응답 시간이 길어질 수 있는 무거운 작업입니다.
* HTTP 요청-응답 모델(Sync)로는 사용자의 대기 시간이 지나치게 길어져 UX가 저해될 우려가 있으며, 서버 자원의 효율적 관리를 위해 비동기(Async) 처리 구조가 필요했습니다.
* 또한, 서로 다른 언어 환경(NestJS - FastAPI) 간의 안정적인 통신과 데이터 정합성을 보장하기 위한 메시지 브로커 도입이 요구되었습니다.

## 🏗️ 2. 기술적 결정 (Decision)
1. **메시지 브로커(RabbitMQ) 도입**: NestJS(Producer)와 FastAPI(Consumer) 간의 결합도를 낮추고 신뢰성 있는 메시지 전달을 위해 RabbitMQ를 채택하였습니다.
2. **Saga 패턴 기반 상태 관리**: 요청 즉시 DB에 `AnalysisTask`를 `PENDING` 상태로 생성하고 `taskId`를 반환합니다. 이후 백그라운드에서 분석이 완료되면 상태를 업데이트하는 Saga 패턴의 첫 단계를 구현했습니다.
3. **Multer Disk Storage 및 수동 유효성 검사**: 서버 메모리 부하를 줄이기 위해 파일을 디스크에 직접 쓰는 `diskStorage`를 사용합니다. 이때 발생한 `file.buffer` 부재 문제는 컨트롤러 수준에서 `mimetype`을 직접 검증하는 방식으로 해결했습니다.
4. **전용 응답 DTO를 통한 데이터 캡슐화**: `AnalysisResponseDto`를 구축하여 유저에게 불필요한 정보(userId 등)를 숨기고, 스웨거(Swagger) 문서를 통해 비동기 작업의 진행 상황을 명확히 전달합니다.

## ⚖️ 3. 결과 (Consequences)
* **장점**: 
    * **확장성**: 분석 엔진(FastAPI)을 별도로 스케일 아웃할 수 있는 유연한 구조를 확보했습니다.
    * **사용자 경험**: 무거운 분석 작업 중에도 사용자는 즉각적인 응답을 받고 다른 기능을 이용할 수 있습니다.
    * **안정성**: `diskStorage` 활용으로 대용량 이미지 처리 시에도 서버 메모리 안정성을 유지합니다.
* **단점**: 
    * RabbitMQ 인프라 관리 비용이 발생하며, 비동기 처리에 따른 최종적 일관성(Eventual Consistency) 관리가 필요합니다.
