# ADR-002: 인증 아키텍처 및 비밀번호 보안 정책 정의

| 항목 | 내용 |
| :--- | :--- |
| **날짜** | 2026-02-27 |
| **상태** | 승인됨 (Accepted) |
| **작성자** | J |
| **영향 범위** | AuthModule / Security Layer |

---

## 🎬 컨텍스트 (Context)
서비스 사용자의 신원을 확인하고 개인화된 기능을 제공하기 위해 인증 시스템이 필요합니다. 초기 설계 단계에서 사용자 정보의 안전한 저장과 효율적인 세션 관리 방식을 결정해야 합니다.

주요 고려 사항:
1. 비밀번호를 DB에 안전하게 저장하는 방법
2. 클라이언트-서버 간 인증 상태 유지 방식 (Stateless vs Stateful)

## 🎀 결정 (Decision)

### 1. Bcrypt 알고리즘 기반 해시 암호화 채택
- 사용자의 비밀번호는 평문으로 저장하지 않으며, `bcrypt` 라이브러리를 사용하여 해시화합니다.
- **선택 이유**: Bcrypt는 솔팅(Salting)과 키 스트레칭(Key Stretching)을 내장하고 있어 레인보우 테이블 공격 및 브루트 포스 공격에 강하며, 연산 비용(Cost)을 조절하여 하드웨어 성능 발전에 대응할 수 있습니다.

### 2. JWT (JSON Web Token) 기반 인증 방식 채택 (F-03 예정)
- 세션/쿠키 방식 대신 토큰 기반 인증 방식을 사용합니다.
- **선택 이유**: 
  - **Stateless**: 서버가 상태를 저장하지 않아 확장성이 뛰어납니다. (특히 향후 MSA 구조인 PetLog 프로젝트 등과의 연계 고려)
  - **Cross-Domain**: 모바일 앱이나 다른 도메인에서도 인증 처리가 유연합니다.

### 3. 입력값 검증 및 비밀번호 복잡도 정책
단순한 비밀번호로 인한 계정 탈취를 방지하기 위해 `class-validator`와 정규표현식(Regex)을 이용한 엄격한 검증 정책을 적용합니다.

#### **비밀번호 검증 규칙**
- **최소 길이**: 8자 이상
- **복잡도 요건**: 영문 대/소문자, 숫자, 특수문자가 각각 최소 1개 이상 포함되어야 함
- **사용 정규식**: `^(?=.*[A-Za-z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$`

#### **DTO 구현 예시 (NestJS)**
```typescript
import { IsEmail, IsNotEmpty, Matches, MinLength } from 'class-validator';

export class LoginDto {
  @IsEmail({}, { message: '유효한 이메일 형식이 아닙니다.' })
  @IsNotEmpty({ message: '이메일은 필수 입력값입니다.' })
  email: string;

  @IsNotEmpty({ message: '비밀번호는 필수 입력값입니다.' })
  @Matches(
    /^(?=.*[A-Za-z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]{8,}$/,
    { message: '비밀번호는 8자 이상이며, 영문, 숫자, 특수문자를 최소 하나씩 포함해야 합니다.' }
  )
  password: string;
}
```

### 🌟 검증 정책 상세 설명 (ADR 기록용)

| 규칙 항목 | 정규식 문법 | 의미 |
| :--- | :--- | :--- |
| **길이 제한** | `{8,}` | 최소 8자 이상의 길이를 가짐 |
| **영문자 포함** | `(?=.*[A-Za-z])` | 최소 하나의 영문 대문자 또는 소문자 포함 |
| **숫자 포함** | `(?=.*\d)` | 최소 하나의 숫자 포함 |
| **특수문자 포함** | `(?=.*[@$!%*?&])` | 지정된 특수문자 중 최소 하나 포함 |

### 4. JWT 상세 설정 (F-03 업데이트)
- **방식**: Bearer Token 인증
- **알고리즘**: HS256
- **페이로드**: 최소한의 식별 정보(`sub`: userId, `email`)만 포함하여 보안 노출 최소화
- **만료 정책**: 보안을 위해 짧은 만료 시간(1시간) 설정

## 🍒 검토한 대안들 (Alternatives Considered)

### 대안 1: Session / Cookie 방식
- **장점**: 서버에서 세션을 즉시 무효화할 수 있어 제어권이 높음.
- **단점**: 서버 메모리 부담이 크고, 서버 확장(Scale-out) 시 세션 클러스터링 등 추가 설정이 필요함.
- **결론**: **기각**. 확장성과 유연성을 위해 JWT를 우선 고려함.

### 대안 2: SHA-256 등 단순 해시 함수
- **단점**: 연산 속도가 너무 빨라 오히려 브루트 포스 공격에 취약함 (솔팅 부재 시 더욱 위험).
- **결론**: **기각**. 보안 수준이 더 높은 Bcrypt를 채택함.

---

## 📸 결과 (Consequences)

### 긍정적 영향
- DB 유출 시에도 비밀번호 원문을 보호할 수 있는 강력한 보안성 확보.
- 서버 부하를 줄이고 유연한 확장이 가능한 인증 구조 구축.
- `class-validator`를 통한 데이터 무결성 보장.

### 부정적 영향 / 주의사항
- **JWT 관리**: 토큰이 탈취될 경우 무효화하기 어렵다는 단점이 있으므로, 추후 Refresh Token 도입 및 만료 시간 설정을 정교하게 해야 함.
- **연산 비용**: Bcrypt 해시 연산은 CPU 집약적이므로 적절한 `saltOrRounds`(예: 10~12) 설정이 필요함.
