## 📋 상태
**Accepted**

## 🎬 1. 배경 (Context)
* NestJS에서 발행한 메뉴 분석 요청(F-05)을 처리하기 위해, OCR 및 AI 라이브러리 생태계가 풍부한 Python 환경의 도입이 필요했습니다.
* 메인 서버와 분석 로직을 분리하여 서비스 간 결합도를 낮추고, 분석 엔진의 독립적인 확장성(Scalability)을 확보하고자 합니다.
* 비동기 메시징 시스템을 완성하기 위해 RabbitMQ로부터 메시지를 수신하고 처리 결과를 관리할 컨슈머(Consumer) 기지가 필요합니다.

## 🏗️ 2. 기술적 결정 (Decision)
1. **FastAPI 프레임워크 채택**: 가볍고 빠른 성능을 제공하며, 비동기(Async/Await) 처리에 최적화된 FastAPI를 분석 엔진의 표준 프레임워크로 선정했습니다.
2. **모노레포(Monorepo) 구조 유지**: 관리 효율성을 위해 `/Project_Diet_Agent` 저장소 내에서 별도의 폴더로 분리하여 관리합니다.
3. **aio-pika를 이용한 비동기 AMQP 통신**: Python의 비동기 특성을 살려 RabbitMQ와 효율적으로 통신하기 위해 `aio-pika` 라이브러리를 사용합니다.
4. **Pydantic 기반 데이터 검증**: NestJS로부터 전달받은 JSON 메시지의 정합성을 보장하기 위해 Pydantic 스키마를 정의하여 런타임 에러를 방지합니다.

## ⚖️ 3. 결과 (Consequences)
* **장점**: 
    * **기술 다양성**: 각 서비스에 가장 적합한 언어(JS/Python)를 사용하여 개발 효율성을 극대화했습니다.
    * **장애 격리**: 분석 엔진에 문제가 생겨도 메인 API 서버의 기능은 유지되는 MSA의 이점을 확보했습니다.
* **단점**: 
    * 두 서비스 간의 데이터 규격(Schema)을 동기화해야 하는 관리 포인트가 추가되었습니다.
