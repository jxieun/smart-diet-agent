# 005: 주간 영양 리포트 집계 및 목표 달성 로직 설계

| **상태** | 승인됨 (Accepted) |

| **영향 범위** | ReportModule / MealModule / Business Logic / DTO |

---

## 🎬 컨텍스트 (Context)
사용자에게 최근 7일간의 영양 섭취 현황을 시각화하여 제공함으로써 데이터 기반의 식단 관리를 돕고자 합니다. 이 과정에서 식단 기록이 없는 날의 시각적 처리 방식과, 사용자의 '목표 칼로리' 대비 성취도를 객관적으로 판단할 수 있는 명확한 비즈니스 로직 수립이 필수적인 과제로 대두되었습니다.

---

## 🎀 결정 (Decision)

### 1. 시계열 데이터의 완전성 보장 (Zero-filling)
- **결정**: DB에 기록이 없는 날짜를 결과에서 제외하는 대신, 서버에서 7일치 빈 객체를 포함한 `Map`을 미리 생성(`dailyMap`)하여 데이터가 없는 날은 `0`으로 채워 반환합니다.
- **이유**: 프론트엔드에서 차트를 렌더링할 때 데이터 배열의 길이를 일정하게 유지하여 클라이언트측 로직을 단순화하고, 사용자에게 '기록 누락' 구간을 명확히 인지시키기 위함입니다.

### 2. 목표 달성(isGoalAchieved) 판단 기준 고도화
- **결정**: 단순히 목표치를 넘었는지가 아니라, 설정된 목표 칼로리의 **90% ~ 110% 범위를 충족**했을 때만 달성(`true`)으로 판정합니다.
- **이유**: 극단적인 절식이나 과식을 모두 지양하고, 계획된 권장 섭취량을 규칙적으로 유지하는 건강한 식습관 형성을 유도하기 위한 비즈니스적 판단입니다.

### 3. 실시간 집계(On-the-fly Aggregation) 및 데이터 격리
- **결정**: 별도의 통계 테이블을 생성하지 않고, API 요청 시마다 Prisma를 통해 최근 7일 데이터를 조회하여 계산하며, 반드시 `userId`를 통한 데이터 격리를 수행합니다.
- **이유**: 현재 데이터 규모에서 실시간 계산 부하가 낮고, 사용자가 기존 식단을 수정/삭제했을 때 리포트에 즉시 반영되는 데이터 정합성을 우선시했기 때문입니다.

---

## 📸 결과 (Consequences)

### 긍정적 영향
- **UX 향상**: 일관된 데이터 구조를 제공하여 차트 구현이 용이해졌으며, 단순 수치 조회를 넘어선 건강 가이드를 제공할 수 있게 되었습니다.
- **데이터 정합성**: 실시간 집계 방식을 통해 사용자의 최신 식단 정보와 리포트 간의 불일치 문제를 원천 차단했습니다.

### 부정적 영향 / 주의사항
- **성능 확장성**: 추후 대규모 트래픽 발생 시 매번 집계 쿼리를 실행하는 것이 DB 부하를 줄 수 있으므로, 향후 Redis를 활용한 캐싱이나 스케줄링 기반의 집계 테이블 도입을 고려해야 합니다.
